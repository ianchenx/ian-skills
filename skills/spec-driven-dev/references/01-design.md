# 阶段 1: 设计讨论

当需求模糊或存在多个可选方案时使用。目标是输出一份设计决策文档。

## 澄清

从 6 个维度快速扫描需求是否清晰：

1. **Objective** — 要达到什么效果？
2. **Done** — 怎么算做完了？
3. **Scope** — 涉及什么？不涉及什么？
4. **Constraints** — 兼容性、性能、依赖限制？
5. **Environment** — 运行时、工具链、部署目标？
6. **Risk** — 会不会破坏现有功能？可以回滚吗？

有 1-2 项不清楚时，先读代码尝试自答 — 能自答就不必打断用户。有 3 项以上不清楚，问 1-5 个关键问题，优先问那些"答案不同会导致完全不同实现路径"的问题。

问题格式上，多选题比开放题更容易回答。提供合理的默认选项，告诉用户可以 `回复 defaults 接受全部默认`。

不要问能通过快速读代码回答的问题 — 那是你的工作，不是用户的。

## 问题定义

清楚地描述三件事：

**现状** — 现在怎么工作的？附代码片段或用户可见的行为。

**问题** — 2-3 句话说清楚哪里不好。带证据（报错信息、用户反馈、或触发路径），因为证据让决策有根据而不是凭感觉。

**期望** — 做完后应该是什么样。用行为描述（outcome），不用技术方案（output），因为描述 outcome 留给实现者自由度，而规定 output 容易锁死在次优方案里。

## 方案设计

### Goals
3-5 个具体的、可衡量的目标。每个目标回答"怎么知道这件事成功了？"

区分用户目标（用户得到什么）和系统目标（系统获得什么能力），因为二者有时候冲突，分开写能让决策更清醒。

### Non-Goals
3-5 件这次明确不做的事 + 每条附上原因（影响不够大 / 太复杂单独做 / 为时过早 / 已有替代方案）。

写 Non-Goals 是因为"什么都想做"是 scope creep 的根源。如果什么都是目标，那就什么都不是目标。

### Candidates
至少列出 2 个方案。对每个方案写清楚：一句话描述、优点、缺点、复杂度评估（简单/中等/复杂）。

只列 1 个方案的"讨论"不是讨论 — 那是通知。至少 2 个选项才能做有意义的取舍。

### Decision
选了哪个方案、为什么选、为什么不选其他的。记录否决方案的理由和选中方案的理由同样重要，因为未来维护者需要知道"当时为什么不那样做"。

### Scope
**v1 范围** — 这次交付什么。具体列出。

**v1 限制** — 这次先不做什么。具体列出，附简短原因。防止实现者过度扩展 scope，也给未来版本留下路标。

### Requirements 分级（可选）
如果需求多，按优先级排：
- **P0 (Must-Have)** — 没有就不能发布
- **P1 (Should-Have)** — 重要但不阻塞
- **P2 (Future)** — 不做但要留空间

## 验证计划

在设计阶段就写好测试计划。测试计划是需求的另一种表达 — 如果写不出怎么验证，说明需求还没想清楚。

## 产出

讨论完成后：

1. **创建或更新 spec 文件**（`docs/specs/NNN-title.md`，`status: design`），包含：问题描述、设计决策（选什么+为什么+否决了什么）、目标。参见 [templates.md](templates.md) 的 Spec 模板。

2. **创建 `design()` issue** — 按 issue tracker 的设计方案模板创建。具体命令参见 issue tracker 命令参考（如 [linear.md](linear.md)）。

3. **问用户：** "设计定了，要继续写完整 spec 吗？" → 进入阶段 2。阶段 2 完成后关闭此 design() issue。
