# 阶段 2: 撰写 Spec

把设计决策或需求固化为结构化的 spec 文档。

**核心哲学：Spec 是发射台，不是脚本。** 写清楚问题和约束，让实现者自己设计方案。

## 好 spec vs 差 spec

好 spec 告诉实现者"去哪"和"不去哪"，但不告诉它"怎么走"。

**好：**
```
## 问题
服务重启后所有运行中的任务信息丢失，用户必须手动重新提交。

## 验收标准
- AC-01: Given 服务被强制终止, When 服务重启, Then 之前的任务被恢复或标记为失败
```

**差（过度规定实现）：**
```
## 执行步骤
1. 创建 persistence.ts，导出 loadState()
2. 在 init() 中调用 loadState()
```

差的 spec 把实现者变成打字员 — 丧失了发现更优方案的机会。

## 流程

1. **理解需求** — 从阶段 1 的 spec draft（status: design）或用户直接描述获取输入
2. **分析代码** — 阅读相关源文件，理解现状和架构约束
3. **撰写 spec** — 按模板补全。如果 spec 文件已存在（status: design），在其基础上补全；否则新建 `<specDir>/NNN-title.md`
4. **更新 status** — 完成后将 frontmatter 改为 `status: spec`
5. 如有 `design()` issue，关闭它（设计决策已落地为 spec）。具体命令参见对应的 `providers/<provider>.md`。

## 写作原则

**写清楚 Why** — 背景和动机比方案重要。如果 spec 里只剩一段话的空间，留给"为什么要做"。

**约束比方案有效** — 告诉实现者什么不能做（不引入新依赖、不改公开 API），约束划定安全边界同时保留设计自由度。

**给需求编 ID** — `REQ-01`、`CON-01`、`AC-01` 让 spec → issue → 代码 → 测试的追踪链条成为可能。三个月后回看时，`REQ-03` 比"那个关于恢复的需求"清晰得多。

**验证方式用场景描述** — 写清楚"做什么 → 预期什么"。GWT 格式可用但非必须 — 核心是让实现者能直接翻译成测试用例。简单场景用自然语言就够。

**设计决策写取舍** — 中等以上复杂度的 spec，写清楚选了什么方案、为什么，以及否决了什么。让实现者理解意图，不会在遇到没写到的情况时做出违背意图的决策。简单需求跳过。

**画出关键模型** — 如果存在状态机、生命周期、数据模型，用文字或 ASCII 图明确描述。实现者看到模型比看到文字描述更快理解系统行为。简单需求跳过。

**写出边界情况** — 异常输入、损坏数据、并发竞态、中途中断 — bug 最常藏身的地方。显式列出来让实现者提前处理。

**列出前置依赖** — 外部 API、权限、前序工作 — 实现者开工前确认是否具备条件，不是做到一半才发现缺东西。

**v1 范围要明确** — "做什么"和"不做什么"同样重要。v1 限制让实现者知道哪里可以简化，让未来维护者知道哪些是刻意推迟的。

**为 AI 实现者优化语言** — 如果读者是 AI agent：用精确术语不用口语化表达，避免歧义词（"处理"可以指解析/转换/存储），让每个段落自包含。Spec 越明确，一次通过率越高。

**测试计划写在 spec 里** — 实现者看到测试计划会知道要写什么测试，也能用测试验证实现是否正确。

**不写代码** — 不写 interface、不写函数签名、不写执行步骤。规定的越多，实现者发现更好方案的概率越低。

**Spec 有生命周期** — spec 不是写完就不管了。实现合并后，回来验证 AC，通过后标记 `status: implemented`。未满足的 AC 创建 follow-up issue。
