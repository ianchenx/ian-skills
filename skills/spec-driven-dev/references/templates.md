# 输出模板

## Spec 文档（完整模板）

```markdown
---
status: design | spec | ready | implemented
---

# SPEC-NNN: [标题]

## 问题
[当前什么坏了 / 缺什么 / 为什么不够好。
写清楚背景，让实现者理解为什么要做。]

## 设计决策（如有）

### Goals
- [用户目标 / 系统目标 — 可衡量的]

### Non-Goals
- [非目标]：[原因]

### Candidates
| 方案 | 优点 | 缺点 | 复杂度 |
|---|---|---|---|
| A: [描述] | [优点] | [缺点] | 简单/中等/复杂 |
| B: [描述] | [优点] | [缺点] | 简单/中等/复杂 |

### Decision
选择方案 [X]：[一句话描述]

**为什么：** [选择理由]

**否决的方案：**
- 方案 [Y]：[为什么不选]

### Open Questions（如有）
- [待解决]（blocking / non-blocking）

## 目标
[做完后世界应该是什么样。
用行为描述，不用技术方案。]

## 约束
- CON-01: [不做什么 / 兼容性要求]
- CON-02: [依赖限制]

## 关键模型（如有）
[状态机、数据模型、生命周期 — 用文字或 ASCII 图描述核心概念的结构。]

## v1 范围
- REQ-01: [这次要做什么]
- REQ-02: [这次要做什么]

## v1 限制
- [先不做什么 -- 附简短原因]

## 验证方式
- [场景描述] → [预期行为]
- [场景描述] → [预期行为]

## 边界情况
- [异常输入/损坏数据] -> [预期行为]
- [并发/竞态/中断场景] -> [预期行为]

## 前置依赖（如有）
- [需要先完成的工作]
- [需要的外部权限 / API]

## 测试计划
- 单元测试：[核心逻辑]
- 集成测试：[跨模块交互]
- 回归测试：[不破坏已有功能]
```

### 各 status 下的最小必填段

| status | 必填段 |
|---|---|
| `design` | 问题、设计决策（含 Candidates + Decision）、目标 |
| `spec` | 以上 + 约束、v1 范围、v1 限制、验证方式、测试计划 |
| `ready` | 同 spec（issues 已创建） |
| `implemented` | 同 ready（验证已通过） |

---

Issue body 模板由 issue tracker 管理。具体模板和命令参见对应的 `providers/<provider>.md`。
